# eBPF-Sentinel 未实现功能可行性与优先级分析

生成日期：2026-05-19

本文基于当前仓库代码、`doc/` 目录文档，以及根目录下 `eBPF-Sentinel-对比分析.md` 中列出的“文档有但仓库缺失/未完全实现”功能进行复核。结论以当前仓库实际代码为准。

## 2. 对比报告提到的未实现功能复核

| 报告提到的功能 | 报告判断 | 当前仓库复核 | 结论 |
|---|---|---|---|
| 进程终止治理 | 未实现 | `process_routes.go` 已提供 `/api/process/kill/:pid` 和 `/api/process/kill/:pid/force`，前端也调用了终止按钮 | 报告已过时，当前已实现 |
| 白名单数据库表 | 未完全实现 | 当前只有网络侧 IP/端口白名单 eBPF Map 和 Go 方法，无 GORM 白名单模型、CRUD API、前端管理入口 | 仍未完整实现 |
| 用户配置表 | 未完全实现 | 告警配置保存在 `AlertPlugin` 内存字段中，进程/网络开关也未落库，重启后恢复默认 | 仍未实现 |
| 系统指标表 | 部分实现/持久化不明确 | 当前持久化 `ExecveEvent`、`NetworkEvent`、`AlertEvent`，没有 `SystemMetric` 表 | 仍未实现 |
| 白名单管理 | 部分吻合 | 网络插件有 `AddIPWhitelist`、`RemoveIPWhitelist`、`AddPortWhitelist`、`RemovePortWhitelist`，但没有 API/UI/持久化；文档中的“可执行路径白名单”未实现 | 仍未完整实现 |
| 策略配置模块 | 部分吻合 | 已有固定的进程/网络开关和告警阈值配置，但没有统一策略模型、策略列表、策略持久化、策略版本或审计 | 仍未完整实现 |
| 安全审计 | 部分吻合 | 告警和关联分析覆盖部分安全检测，但没有独立审计日志、审计查询、审计报表、审计策略管理 | 仍未完整实现 |
| CPU 异常波动检测 | 部分吻合 | 当前主要是 CPU 阈值告警，不是基于趋势/基线/突变的异常波动检测 | 仍未完整实现 |

## 3. 未实现或未完整实现功能的可行性与优先级

优先级定义：

- P0：影响文档承诺或核心产品闭环，应优先补齐。
- P1：明显增强可用性、可运维性或安全分析能力，建议第二阶段实现。
- P2：价值存在，但实现复杂度或风险较高，可放入后续规划。

| 功能 | 当前缺口 | 可行性 | 建议优先级 | 原因 |
|---|---|---|---|---|
| 用户配置持久化 | 告警阈值、监控开关重启后丢失 | 高 | P0 | 已有 API 和内存配置结构，只需新增 GORM 模型、初始化加载、更新落库；成本低，直接提升可用性 |
| 白名单持久化与管理 API | IP/端口白名单只停留在插件方法和 eBPF Map 层，没有 API/UI/数据库 | 高 | P0 | 网络插件已经具备 Map 更新能力，补齐 `WhitelistRule` 表、CRUD 路由和启动同步即可形成闭环 |
| 可执行路径白名单 | 文档描述的是“受信任可执行路径”，但当前 execve 没有路径白名单过滤 | 中 | P0 | 需要在用户态基于 execve 事件过滤或告警抑制，若下沉到 eBPF 需处理字符串 Map 和路径长度问题；作为文档承诺功能优先级高 |
| 系统指标持久化 | CPU/内存/网速只实时展示，没有历史表 | 高 | P1 | `system` 事件已在事件总线中流转，新增 `SystemMetric` 表并按周期写入即可；需控制写入频率和保留策略 |
| 统一策略配置模块 | 目前策略散落在 `policy.go`、`AlertPlugin` 和网络插件白名单方法中 | 中 | P1 | 需要抽象 `PolicyStore`/`PolicyService`，统一监控开关、阈值、白名单和采样参数；有助于后续扩展，但涉及面较广 |
| 完整安全审计模块 | 只有告警事件，没有审计事件分类、查询、导出、处置记录 | 中 | P1 | 可复用现有事件和告警表先做审计视图，再逐步新增 `AuditLog`；对安全产品定位有明显价值 |
| CPU 异常波动检测 | 当前阈值告警无法识别突增、突降、持续异常、历史基线偏离 | 中 | P1 | 可以先用滑动窗口均值/方差/EWMA 实现轻量波动检测；需要一定测试避免误报 |
| 关键系统调用过滤 | 文档提到系统调用级别监控和过滤，当前主要覆盖 execve 与网络 TC | 中低 | P2 | 需要新增多个 tracepoint/kprobe 或 LSM/eBPF 方案；如果涉及“阻断”而非“监控”，权限、兼容性和安全风险更高 |
| 白名单前端管理界面 | 设置页没有白名单管理入口 | 高 | P1 | 后端白名单 API 完成后，前端增加 IP/端口/路径规则表单和列表即可；依赖白名单 API |
| 配置变更审计 | 策略、阈值、白名单变更没有审计记录 | 高 | P1 | 可在写接口统一记录 `AuditLog`，实现成本低；对远程管理和答辩展示都有价值 |

## 4. 推荐实现顺序

### 第一阶段：补齐文档承诺闭环（P0）

1. 新增 `UserConfig` 或 `AlertConfigRecord` 模型，把告警阈值和监控开关持久化。
2. 新增 `WhitelistRule` 模型，支持 `type=ip|port|exec_path`、`value`、`enabled`、`created_at`。
3. 为网络 IP/端口白名单提供 CRUD API，并在启动时从数据库同步到 eBPF Maps。
4. 为 execve 事件增加可执行路径白名单处理。建议先在用户态做过滤或告警抑制，降低 eBPF 字符串处理复杂度。

### 第二阶段：增强策略与审计能力（P1）

1. 建立统一策略服务，把监控开关、告警阈值、白名单、采样参数纳入同一配置入口。
2. 新增系统指标历史表，支持最近一段时间 CPU、内存、网络速率查询。
3. 新增配置变更审计日志，记录操作者、来源 IP、操作类型、变更前后内容。
4. 前端设置页增加白名单管理、策略列表和审计查询入口。

### 第三阶段：高级检测能力（P1/P2）

1. 在 `AlertPlugin` 中加入 CPU 波动检测规则，例如滑动窗口均值偏离、短时间突增、持续高位。
2. 扩展安全审计报表，按进程、网络连接、告警、处置状态聚合查询。
3. 评估关键系统调用监控范围，例如文件访问、权限变更、进程注入相关系统调用。
4. 如需实现“过滤/阻断”级能力，应单独设计权限边界、误杀恢复和兼容性测试。

## 5. 风险与注意事项

| 风险点 | 说明 | 建议 |
|---|---|---|
| eBPF 字符串过滤复杂 | 可执行路径白名单如果直接在 eBPF 中做，受 Map key 长度、Verifier、路径获取可靠性影响 | 先在用户态实现过滤/告警抑制；必要时再下沉 |
| 白名单语义需明确 | 当前网络白名单逻辑是“启用后只监控白名单内 IP/端口”，不是“放过白名单不告警” | 实现 API 前先命名清楚，可区分 monitor allowlist 与 alert allowlist |
| 配置持久化一致性 | 数据库、内存配置、eBPF Map 三者可能不一致 | 引入统一配置加载和同步流程，API 更新成功后再返回 |
| 进程治理安全性 | 终止进程已实现，但属于高风险操作 | 继续保留 loopback/token 限制，并增加审计日志 |
| 系统指标写入量 | 高频写入 SQLite 会带来膨胀和锁竞争 | 设置采样周期、保留期限和批量清理策略 |

## 6. 总结

当前仓库的核心监控、实时推送、告警、进程治理和基础策略开关已经较完整。`eBPF-Sentinel-对比分析.md` 中“进程终止治理未实现”的判断已不符合当前代码状态。

仍需重点补齐的是配置和管理闭环：用户配置持久化、白名单数据库/API/UI、可执行路径白名单、系统指标历史表、统一策略模块和审计日志。这些功能整体可行性较高，建议优先按 P0/P1 顺序推进。
