# eBPF-Sentinel 文档功能与仓库实际功能对比分析报告

## 一、文档中描述的系统框架

根据答辩稿《基于eBPF的Linux系统行为监控与可视化框架的设计与实现》，文档描述的系统架构如下：

### 1.1 技术栈

| 层次 | 技术选型 |
|------|----------|
| 内核侧 | C 语言编写 eBPF 程序，clang/LLVM 编译为 eBPF bytecode，CO-RE 解决跨内核版本兼容性 |
| 后端 | Go 语言，Cilium eBPF-go 库（cilium/ebpf），Gin Web 框架 |
| 前端 | 原生 HTML/CSS/JavaScript，WebSocket 实时推送 |
| 数据库 | SQLite，GORM 自动迁移 |
| 部署 | B/S 结构，静态编译单二进制文件 |

### 1.2 系统功能模块（文档表3-1）

| 模块分类 | 功能 |
|----------|------|
| **监控模块** | 进程监控、网络监控、系统指标查看 |
| **管理模块** | 监控配置、白名单管理、策略设置 |

### 1.3 文档中描述的具体功能

| 功能 | 文档描述 |
|------|----------|
| **进程生命周期监控** | 通过 tracepoint 挂载 execve 系统调用，采集 PID、PPID、进程名、程序路径 |
| **关键系统调用过滤** | 系统调用级别的监控和过滤 |
| **CPU 异常波动检测** | 实时采集 CPU 使用率，异常告警 |
| **异常网络连接捕获** | TC 挂载网络数据包，捕获网络连接信息 |
| **安全审计** | 对特定进程行为做安全审计 |
| **白名单管理** | 受信任的可执行路径管理 |
| **进程治理** | **进程终止治理**功能 |
| **监控开关控制** | 动态启停监控模块 |
| **策略配置** | 告警阈值等配置管理 |
| **WebSocket 实时推送** | 实时数据推送到前端 |

### 1.4 文档中描述的数据库实体

| 实体 | 说明 |
|------|------|
| 进程事件 | execve 系统调用的信息 |
| 网络事件 | 数据包的地址和端口 |
| 系统指标 | CPU 使用率、内存占用量 |
| 白名单 | 受信任的可执行路径 |
| 用户配置 | 告警阈值 |

## 二、仓库中实际实现的功能

### 2.1 eBPF 内核程序（C 代码）

| 文件 | 挂载点 | 功能描述 |
|------|--------|----------|
| `ebpf/execve.c` | `tracepoint/syscalls/sys_enter_execve` | 进程 execve 监控，采集 PID、PPID、进程名(argv0)、程序路径(comm)，通过 Ring Buffer 发送到用户态，支持全局开关控制 |
| `ebpf/network.c` | `TC ingress/egress` | 网络流量监控，解析以太网/IP/TCP/UDP/ICMP 头部，采集五元组、包大小、方向、进程名，采样率 1/100，可疑端口绕过采样，支持 IP/端口白名单过滤和全局开关 |
| `ebpf/cpu.c` | `tracepoint/sched/sched_switch` | CPU 使用率统计，通过 busy_ns/idle_ns 计算，Per-CPU Array 存储 |

### 2.2 Go 后端插件

| 插件 | 文件 | 功能描述 |
|------|------|----------|
| **ExecvePlugin** | `internal/plugin/execve_plugin.go` | 加载 execve eBPF 对象，挂载 tracepoint，Ring Buffer 读取事件 |
| **NetworkPlugin** | `internal/plugin/network_plugin.go` | 遍历网络接口，挂载 TC ingress/egress 程序 |
| **CPUPlugin** | `internal/plugin/cpu_plugin.go` | 加载 cpu eBPF 对象，读取 CPU 统计，计算 CPU 使用率 |
| **SystemMonitorPlugin** | `internal/plugin/system_plugin.go` | 使用 gopsutil 采集系统指标（CPU、内存、网络速率），周期性采集 |
| **AlertPlugin** | `internal/plugin/alert_plugin.go` | 告警派生引擎，包含多种告警规则 |

### 2.3 AlertPlugin 告警规则详情

| 告警类型 | 说明 |
|----------|------|
| **CPU 阈值告警** | CPU 使用率超过阈值触发（默认 85%） |
| **内存阈值告警** | 内存使用率超过阈值触发（默认 90%） |
| **网络速度告警** | 网络速度超过阈值触发（默认 10240 KB/s） |
| **大包告警** | 单包大小超过阈值触发（默认 1MB） |
| **可疑端口告警** | 检测到连接可疑端口（22, 23, 3389, 4444, 5555, 31337）触发 |
| **敏感命令告警** | 执行敏感命令触发 |
| **数据外泄检测** | 检测大流量出站连接，可能的数据外泄行为 |
| **关联分析告警** | 基于滑动窗口的多事件关联分析（correlation） |
| **告警冷却** | 支持冷却时间防止告警风暴 |
| **告警状态** | active/resolved 状态跟踪 |

### 2.4 API 端点

| 方法 | 路径 | 功能 |
|------|------|------|
| GET | `/api/events` | 获取最近 100 条进程事件 |
| GET | `/api/network-events` | 获取最近 100 条网络事件 |
| GET | `/api/alerts` | 获取告警事件（支持 history 参数） |
| PATCH | `/api/alerts/:id/status` | 更新告警状态 |
| GET | `/ws` | WebSocket 实时数据推送 |
| 相关 | `/api/policy/*` | 策略开关控制（进程/网络监控启停） |

### 2.5 前端功能（web/dist/index.html）

| 功能区域 | 具体内容 |
|----------|----------|
| **统计面板** | 总事件数、进程事件、网络事件、告警事件、CPU 使用率、内存使用率、下载速度、上传速度、每秒事件、运行时间 |
| **全部事件视图** | 实时事件列表，最多显示 500 条 |
| **进程监控视图** | btop 风格表格，显示 PID、CPU%、MEM%、RSS、NAME，支持搜索/排序 |
| **网络事件视图** | 五元组信息表，网速趋势图（Canvas 绘制 60 秒趋势），支持搜索过滤 |
| **告警中心** | 告警列表，显示级别、时间、规则、来源、消息、状态，支持操作 |
| **监控开关** | 进程监控开关、网络监控开关（设置面板中） |
| **告警阈值配置** | CPU 阈值、内存阈值、网络速度阈值、单包大小限制、冷却时间、关联窗口、关联最大间隔、外泄阈值、单指标告警开关 |

### 2.6 仓库中额外的功能

| 功能 | 说明 |
|------|------|
| **Token 认证** | `security.go` 中实现了基于 Token 的认证中间件 |
| **进程路由** | `process_routes.go` 提供进程列表查询（含 CPU/内存使用率） |
| **关联分析引擎** | `correlation.go` 实现多事件关联检测 |
| **告警单元测试** | `alert_plugin_test.go` 告警插件测试 |
| **前端编译产物** | 使用 `app.css` 和 `app.js`，前端有构建流程 |

## 三、文档功能与仓库功能对比分析

### 3.1 完全吻合的功能

| 功能 | 文档描述 | 仓库实现 | 吻合度 |
|------|----------|----------|--------|
| **进程监控** | 通过 tracepoint 采集 execve 事件，获取 PID、PPID、进程名、程序路径 | `ebpf/execve.c` 实现 tracepoint 挂载，`ExecvePlugin` 处理事件 | **完全吻合** |
| **网络监控** | TC 挂载，捕获网络数据包 | `ebpf/network.c` TC ingress/egress 挂载，`NetworkPlugin` 处理 | **完全吻合** |
| **系统指标查看** | CPU 使用率、内存占用量 | `SystemMonitorPlugin` + `CPUPlugin` 采集 CPU、内存、网络速率 | **完全吻合** |
| **WebSocket 实时推送** | 前端通过 WebSocket 长连接实时接收数据 | `internal/websocket` 包实现 Hub 管理，`/ws` 端点 | **完全吻合** |
| **监控开关控制** | 策略配置中可控制监控启停 | `policy.go` 实现 execve/network 监控开关，eBPF Map 控制 | **完全吻合** |
| **SQLite 数据库** | 使用 SQLite 存储，GORM 自动迁移 | `internal/models` 包实现，CreateEvent/CreateNetworkEvent 等 | **完全吻合** |
| **Ring Buffer 传输** | 使用 Ring Buffer 进行内核态到用户态数据传输 | `ebpf/execve.c` 和 `ebpf/network.c` 均使用 BPF_MAP_TYPE_RINGBUF | **完全吻合** |

### 3.2 部分吻合的功能

| 功能 | 文档描述 | 仓库实现 | 差异分析 |
|------|----------|----------|----------|
| **白名单管理** | 独立的管理模块，受信任的可执行路径 | 网络 eBPF 代码中有 IP 白名单和端口白名单的 Map 定义（`ip_whitelist`、`port_whitelist`），但前端**没有专门的白名单管理界面**，API 中没有白名单 CRUD 接口 | **部分吻合** - 内核侧有白名单机制，用户态管理界面缺失 |
| **策略配置** | 管理模块中的策略设置 | 前端有告警阈值配置界面，支持 CPU/内存/网络等阈值设置，但**没有独立的策略配置模块** | **部分吻合** - 告警阈值配置实现，整体策略配置功能不完整 |
| **安全审计** | 对特定进程行为做安全审计 | `AlertPlugin` 实现了敏感命令告警、可疑端口检测、数据外泄检测等安全相关功能，但**没有完整的安全审计模块** | **部分吻合** - 告警功能覆盖部分安全审计需求 |
| **CPU 异常波动检测** | 检测 CPU 异常波动 | `AlertPlugin` 支持 CPU 阈值告警（默认 85%），但**没有波动趋势分析**功能 | **部分吻合** - 简单阈值告警而非异常波动检测 |

### 3.3 文档中有但仓库中缺失的功能

| 功能 | 文档描述 | 仓库实现状态 | 说明 |
|------|----------|-------------|------|

## 四、总体评估

### 4.1 架构吻合度

| 维度 | 吻合度 | 说明 |
|------|--------|------|
| 整体技术栈 | **95%** | eBPF(C) + Go + Gin + SQLite + WebSocket 架构完全一致，仅前端构建流程文档未提及 |
| 内核侧实现 | **90%** | 三个 eBPF 程序（execve、network、cpu）与文档描述一致，额外实现了采样、白名单等增强功能 |
| 后端实现 | **85%** | 核心插件（Execve、Network、CPU、System）吻合，额外增加了 AlertPlugin 告警引擎 |
| 前端实现 | **80%** | 基本页面和实时推送吻合，前端界面比文档描述更丰富（告警中心、网速趋势图、设置面板） |
| 数据库设计 | **70%** | 核心事件表吻合，但白名单表和用户配置表未完整实现 |

### 4.2 功能模块吻合度

| 文档模块 | 文档功能 | 仓库实现 | 吻合度 |
|----------|----------|----------|--------|
| **监控模块-进程监控** | execve 事件采集 | 完整实现 | **100%** |
| **监控模块-网络监控** | 网络流量捕获 | 完整实现，额外有采样和白名单 | **100%** |
| **监控模块-系统指标** | CPU、内存查看 | 完整实现 | **100%** |
| **管理模块-监控配置** | 监控开关 | 实现（eBPF Map + API） | **80%** |
| **管理模块-白名单管理** | 可执行路径白名单 | 网络侧有 IP/端口白名单，无管理界面 | **40%** |
| **管理模块-策略设置** | 策略配置 | 仅实现告警阈值配置 | **50%** |
| **文档提及但未分类** | 进程终止治理 | **未实现** | **0%** |

### 4.3 结论

**eBPF-Sentinel 仓库代码与答辩稿文档的整体吻合度约为 80%。**

核心监控功能（进程、网络、系统指标）实现与文档描述高度一致，技术架构完全吻合。主要差异点在于：

1. **文档中缺失描述仓库的重要功能**：告警关联分析引擎、数据外泄检测、可疑端口检测、告警状态跟踪、Token 认证等是仓库的重要功能但文档完全没有提及。

2. **文档描述但仓库未实现的功能**：进程终止治理功能是文档明确列出的用例但实际代码中完全缺失；白名单管理功能内核侧有实现但用户态管理界面缺失。

3. **功能增强**：仓库实际功能比文档描述的更加丰富，特别是在安全告警方面（AlertPlugin 的多维度告警和关联分析），前端界面也更加完善。

建议答辩时以仓库实际代码为准，补充文档中遗漏的功能描述，并说明进程终止治理等功能的后续实现计划。
